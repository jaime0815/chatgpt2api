# 普通用户 ChatGPT 式聊天界面设计

日期：2026-07-10

## 1. 背景

当前普通用户登录后默认进入 `/image`，只能使用正式的图片生成与编辑工作台。文本聊天仅存在于管理员 `/debug` 页的调试面板中；该面板没有流式输出、正式会话历史、Markdown 渲染、停止生成、重试、编辑消息或用户级本地隔离。

后端已经提供普通用户可访问的模型、文本聊天、图片生成、图片编辑和图片任务 API，但没有通用 PDF/Office 文档附件能力，也没有服务端聊天会话 CRUD。

本设计新增独立 `/chat` 功能层，为普通用户提供接近 ChatGPT 的正式聊天体验，同时保留 `/image` 作为高级画图工作台。

## 2. 目标

1. 普通用户登录后默认进入 `/chat`。
2. 提供经典 ChatGPT 布局：桌面固定会话侧栏，移动端历史抽屉，主区为消息流和底部输入框。
3. 支持动态聊天模型选择、流式输出、停止生成、重新生成、编辑用户消息后重发、复制回答和 Markdown/代码块渲染。
4. 支持图片、PDF 和常见 Office/文本文件附件。
5. 支持在聊天中显式开启“生成图片”，复用现有图片任务、参考图及全部图片参数。
6. 按普通用户 `subjectId` 在浏览器本地保存会话、附件和参数，不同用户之间完全隔离。
7. 保持现有 OpenAI 兼容接口和高级画图工作台行为不变。
8. 使用多个 agent 在独立 git worktree 中并行实现，每个任务测试通过后分别提交，最终保留独立提交历史。

## 3. 非目标

- 不实现服务端聊天历史、跨浏览器或跨设备同步。
- 不实现会话分享链接、会话分支、历史搜索或导出。
- 不支持旧版二进制 Office 格式 `DOC`、`XLS`、`PPT`。
- 不新增本地 PDF/Office 文档解析 fallback；文档由 ChatGPT 上游原生读取。
- 不把管理员调试页直接改造成普通用户正式页面。
- 不替换或删除现有 `/image` 高级画图工作台。
- 不在本期引入通用工具调用、联网搜索或文件生成能力。

## 4. 用户入口与权限

### 4.1 路由

- 新增 `/chat`。
- 普通用户默认路由从 `/image` 改为 `/chat`。
- `/image` 保留，并在聊天侧栏中显示为“高级画图”。
- `/chat` 对已认证的 `user` 和 `admin` 均可访问；只有普通用户的默认落点发生变化。
- 管理端现有 `/accounts`、`/logs`、`/debug`、`/settings` 等权限保持不变。

### 4.2 应用壳层

当前根布局始终渲染 `TopNav`、页面内边距和最大宽度。经典聊天页需要沉浸式全高布局，因此新增路由感知的应用壳层：

- `/chat` 隐藏现有 `TopNav`，由聊天侧栏负责品牌、聊天、高级画图、用户信息和退出入口。
- 其他页面继续使用现有壳层和 `TopNav`，视觉和尺寸不变。
- 所有路由和资源继续遵守 `NEXT_PUBLIC_BASE_PATH`，使用 `Link`、`withBasePath`、`getApiBaseUrl` 或现有请求封装，不硬编码根路径。

## 5. 界面设计

### 5.1 桌面端

- 左侧固定约 248px 会话侧栏。
- 侧栏顶部依次为品牌、新对话、聊天、高级画图。
- 会话按“今天”“过去 7 天”“更早”分组，支持选择、重命名和删除。
- 侧栏底部显示当前用户名称、角色、主题和退出入口。
- 主区顶部只保留聊天模型选择和当前会话操作。
- 消息内容最大宽度约 760px；输入框最大宽度约 780px 并固定在底部。
- 用户消息使用紧凑气泡；助手消息保持开放排版，便于阅读 Markdown、代码块和列表。

### 5.2 移动端

- 固定侧栏隐藏，由左上角菜单打开左侧抽屉。
- 顶部居中显示当前聊天模型，右侧提供新对话命令。
- 输入框固定在安全区域上方。
- 图片设置使用底部抽屉，避免小屏浮层溢出。
- 所有按钮、附件卡片和长文件名必须在窄屏下稳定换行或截断，不得产生横向页面滚动。

### 5.3 空会话

- 不展示营销页或功能说明卡片。
- 中央仅显示简洁的问候/任务提示和输入框，用户可直接开始工作。
- 输入框占据主要视觉焦点。

## 6. 核心聊天交互

### 6.1 发送与流式输出

- `Enter` 发送，`Shift+Enter` 换行。
- 发送后立即写入用户消息和一个 `streaming` 助手占位消息。
- 使用 Fetch API 读取 SSE，逐块追加 `choices[0].delta.content`。
- SSE 客户端必须识别 HTTP 成功后的流内 `{"error": ...}`，不能只依赖 HTTP 状态。
- `error`、`stopped` 和 `complete` 都是终态。收到流内 `error` 后立即标记 `error`；现有后端即使随后继续发送 `[DONE]`，客户端也必须忽略该 `[DONE]`，不得把错误覆盖为 `complete`。
- 只有当前消息仍处于 `streaming` 时，收到 `[DONE]` 才把助手消息标记为 `complete`。

### 6.2 停止、失败与重试

- 流式生成期间发送按钮变为停止按钮，调用 `AbortController.abort()`。
- 浏览器断开后，ASGI 流式生成器必须捕获取消/断连，在 `finally` 中关闭上游响应迭代器和 `OpenAIBackendAPI` session，并释放本次账号资源。停止不能只是不再渲染而让上游继续占用账号。
- 停止后保留已生成文本，并将消息标记为 `stopped`。后续服务端错误或 `[DONE]` 不得覆盖该状态。
- 上传或生成失败时保留原用户消息、附件和输入参数，将对应助手消息标记为 `error`。
- “重新生成”使用该助手消息之前的历史重新请求，不重复添加用户消息。
- 上游账号只能在首个输出 token 之前自动切换；已经产生输出后失败时不静默重试，避免重复或冲突回答。

### 6.3 编辑消息

- 用户可编辑自己的历史消息。
- 保存编辑并重发时，删除该消息之后的所有分支消息，然后从编辑后的消息重新生成。
- 被编辑消息引用的附件保持关联；用户可以在重发前移除或新增附件。

### 6.4 消息操作

- 助手消息提供复制、重新生成、赞和踩的图标操作。
- 用户消息提供编辑和复制。
- Markdown 使用现有 `react-markdown` 与 `remark-gfm`。
- 代码块提供复制按钮和横向内部滚动，不扩大页面宽度。
- 未知或不安全 HTML 不直接执行。

## 7. 模型选择

- 页面加载后调用现有 `GET /v1/models`。
- 聊天模型选择器过滤所有图片模型，默认使用 `auto`。
- 图片模型识别逻辑集中为共享 helper，不在多个组件中重复字符串判断。
- 当前选择按 `subjectId` 保存；模型已从服务端列表消失时明确提示并回到 `auto`，不静默替换为其他具体模型。
- 图片模型不出现在顶部聊天模型选择器中；图片模型只在“图像设置”内选择。

## 8. 附件能力

### 8.1 支持格式

图片：

- PNG
- JPEG/JPG
- WEBP
- GIF

文档：

- PDF
- DOCX
- XLSX
- PPTX
- TXT
- MD
- CSV

不支持 `DOC`、`XLS`、`PPT`，也不对不支持格式做静默文本转换。

### 8.2 限制

- 单次消息最多 10 张图片，每张最大 10 MB。
- 单次消息最多 5 个文档，每个最大 25 MB。
- 单次消息新增加的全部附件总量最大 50 MB。
- 单个会话当前仍被历史消息引用的唯一附件工作集最大 100 MB。每次 multipart 请求只携带请求消息实际引用的去重附件，并执行同样的 100 MB 请求上限。
- 新增附件会使会话工作集超过 100 MB 时，前端阻止发送并提示用户新建对话，或编辑较早的用户消息并截断其后的历史以释放附件引用；后端再次执行最终校验。
- 生图的“生成数量”继续完整沿用现有图片工作台参数，可使用当前 1-100 的任务拆分行为；它不受“最多 10 张输入/参考图片”限制。
- 前端用于即时反馈，后端执行同样的扩展名、MIME、数量、大小和总量校验，后端结果为最终判定。

### 8.3 选择与展示

- 输入框“+”菜单提供图片和文件选择。
- 支持点击、拖放和粘贴图片；文档使用点击或拖放。
- 待发送附件以可移除卡片展示，图片显示缩略图，文档显示类型、文件名和大小。
- 文件名过长时截断但保留完整 tooltip。
- 同一个本地文件通过 SHA-256 去重。

### 8.4 上游原生文件读取

后端新增通用文件上传适配器，复用现有三段上游协议：

1. `POST /backend-api/files` 获取 `upload_url` 和 `file_id`。
2. 使用签名 URL `PUT` 上传原始字节。
3. `POST /backend-api/files/{file_id}/uploaded` 确认完成。

图片上传继续包含宽高和 `image_asset_pointer`。普通文档的上游请求字段不能凭图片结构推测，必须先完成一次当前 ChatGPT Web 的真实协议探测，并把脱敏后的请求/响应保存为测试 fixture。fixture 至少覆盖文件创建、上传确认和 conversation message 中的文档 content part、metadata attachment 以及上游处理状态。

内部接口在协议探测前固定为：

- `upload_chat_attachment(data, file_name, mime_type) -> UploadedChatAttachment`
- `UploadedChatAttachment` 包含内部稳定字段：文件 ID、文件名、MIME、大小、asset pointer、conversation content part 和 metadata attachment。
- Web 聊天编排只依赖这个内部结构，不直接拼接上游专有字段。

上游通用附件适配器只有在脱敏 fixture、单元测试和至少一个真实 PDF 请求同时通过后才算完成。如果没有可用真实账号，该任务必须报告阻塞，不能把文档附件能力声明为完成。

文件必须由最终发送对话的同一个上游账号上传。前端不会获得或保存裸上游 `file_id`。

## 9. Web 聊天接口

### 9.1 新接口

新增 Web 专用 `POST /api/chat/stream`，不改变现有 `/v1/chat/completions` 兼容接口。

请求使用 `multipart/form-data`：

- `request`：序列化后的聊天请求 JSON。
- `files`：所有本次请求消息实际引用的唯一附件 Blob，顺序与请求中的附件 manifest 一致，总量不得超过 100 MB。

请求 JSON 包含：

- `model`
- `messages`
- 每条消息的稳定 ID、角色、文本和附件 ID
- 附件 manifest：稳定 ID、文件名、MIME、大小、SHA-256
- 可选的聊天参数

后端必须核对 manifest 与实际文件数量、顺序、大小和 SHA-256，拒绝缺失或不匹配的附件。

### 9.2 后端编排

单次请求顺序为：

1. 验证身份、文本和附件。
2. 选择文本上游账号。
3. 对当前账号解析或上传所有唯一附件。
4. 根据附件类型构造上游 conversation messages。
5. 使用同一个账号发起 SSE 对话。
6. 仅在首个 delta 之前允许刷新或更换失效账号；换账号后重新解析附件。
7. 输出 OpenAI `chat.completion.chunk` 兼容 SSE 和最终 `[DONE]`。

服务端可维护仅内存的短期附件引用缓存，键为“上游账号指纹 + 文件 SHA-256”。缓存只保存上游附件元数据，不保存原始文档；引用失效时重新上传。

### 9.3 内容过滤与日志

- 继续执行现有文本内容过滤。
- 原始文件内容不写入应用日志。
- 日志只允许记录文件名、MIME、大小、哈希前缀、上游阶段和错误摘要。
- 不记录裸访问令牌、签名上传 URL 或裸上游 `file_id`。

## 10. 图片生成模式

### 10.1 触发方式

- 输入框“+”菜单提供显式“生成图片”命令。
- 开启后显示可移除的“生成图片”模式标签。
- 纯文本提示且无参考图时调用现有图片生成任务。
- 存在参考图片时调用现有图片编辑任务。
- 文档附件不能与图片生成模式同时发送；前端给出明确提示。

### 10.2 复用现有参数

聊天页和 `/image` 共用同一份图片设置定义与校验：

- 图片模型
- 质量：自动、低、中、高
- 自定义宽度和高度
- 宽高比预设
- 1k、2k、4k 档位
- 生成数量及自定义数量
- Codex 图片模型对 2k/4k 的现有限制
- 剩余额度和进行中任务数

桌面端点击参数摘要打开设置浮层；移动端打开底部抽屉。参数摘要示例为“高 · 16:9(2k) · 3 张”。

### 10.3 任务与消息

- 每轮图片消息保存完整参数快照。
- 后续修改设置只影响下一轮，不改写历史参数。
- 继续复用现有图片任务 ID、轮询、超时继续等待、失败重试和刷新恢复逻辑。
- 结果直接插入当前聊天时间线，支持预览、下载、再次生成和作为下一轮参考图。
- 从历史图片轮次再次生成时恢复该轮的模型、质量、宽高、比例、档位、数量和参考图。
- 聊天历史只持久化图片的服务端 URL、task ID、状态、尺寸、修订提示和参数快照，明确丢弃任务响应中的 `b64_json`。作为参考图继续时再通过服务端 URL 获取 Blob。

## 11. 本地会话存储

### 11.1 命名空间

新增独立的聊天会话和附件存储，不复用图片会话固定 `items` 键。

- 所有键以 `subjectId` 分区。
- 切换登录用户后只能加载当前用户数据。
- 用户可清空自己的本地聊天历史。

### 11.2 数据结构

会话至少保存：

- 会话 ID、标题、创建和更新时间
- 消息列表和消息状态
- 当前聊天模型
- 文本、附件引用、图片任务引用
- 图片生成参数快照
- 每个会话的滚动位置

图片消息不得把 `b64_json` 写入聊天 IndexedDB。现有图片任务同时返回 URL 和 Base64 时，聊天数据层只提取 URL 与必要元数据。

附件单独保存：

- 附件 ID
- `subjectId`
- SHA-256
- 文件名、MIME、大小
- Blob
- 引用计数或可重建的引用关系

删除会话时清理不再被任何消息引用的 Blob。渲染图片时按需创建并释放 Object URL。

### 11.3 容量失败

- IndexedDB 配额不足时当前会话继续保存在内存中。
- 页面明确提示“当前对话无法写入本地历史”。
- 不因持久化失败清空输入、消息或附件。

建议结构：

- `web/src/app/chat/page.tsx`：路由入口和数据装配。
- `web/src/app/chat/components/chat-shell.tsx`：桌面/移动应用壳层。
- `web/src/app/chat/components/chat-sidebar.tsx`：会话历史和用户入口。
- `web/src/app/chat/components/chat-thread.tsx`：消息列表和滚动管理。
- `web/src/app/chat/components/chat-message.tsx`：文本、附件和图片消息。
- `web/src/app/chat/components/chat-composer.tsx`：输入、附件和模式切换。
- `web/src/app/chat/components/chat-model-select.tsx`：聊天模型选择。
- `web/src/app/chat/components/chat-attachment-picker.tsx`：附件选择与验证。
- `web/src/app/chat/lib/chat-stream.ts`：SSE 读取与错误解析。
- `web/src/app/chat/lib/chat-models.ts`：模型分类。
- `web/src/store/chat-conversations.ts`：会话和附件持久化。
- 共享图片设置模块：由现有 `ImageComposer` 提取参数定义、标签、校验和设置面板，两页共同使用。

页面组件只负责组合；流协议、存储、附件校验和图片参数不堆积在单个 `page.tsx` 中。

## 13. 错误状态

- `draft`：本地草稿。
- `sending`：请求和附件上传中。
- `streaming`：助手流式生成中。
- `complete`：完成。
- `stopped`：用户停止，保留部分内容。
- `error`：失败，可重试。
- `queued/running`：图片任务排队或执行中。

错误文案优先使用后端结构化消息，并为以下场景提供明确说明：

- 不支持的文件类型
- 文件数量或大小超限
- 文档与生图模式冲突
- 上游附件上传失败
- 模型不可用
- 无可用文本账号或图片额度
- SSE 中断
- 本地存储配额不足

## 14. 测试设计

### 14.1 后端自动化测试

- Web 聊天接口身份验证。
- multipart manifest 与文件匹配。
- 支持格式、MIME、扩展名、数量和大小边界。
- 最多 10 张图片输入。
- 同一个账号完成附件上传和消息发送。
- 首个 token 前账号失效时换账号并重新上传。
- 首个 token 后失败不自动重放。
- SSE 正常 delta、流内错误和 `[DONE]`。
- 流内 error 后的 `[DONE]` 不得覆盖 `error`；停止后的任何后续事件不得覆盖 `stopped`。
- 客户端断连会关闭上游迭代器和 backend session，不继续占用账号。
- 单条消息 50 MB 与会话/请求附件工作集 100 MB 边界。
- 原始文件、令牌和上游 file ID 不进入日志。
- 现有 `/v1/chat/completions`、图片任务和模型接口回归测试继续通过。

### 14.2 前端自动化测试

引入 Vitest 和 React Testing Library，按 TDD 覆盖：

- SSE chunk、跨 chunk 行、流内错误和 `[DONE]` 解析。
- 模型列表分类和失效模型处理。
- 按 `subjectId` 的会话隔离。
- 附件 SHA-256 去重、引用和删除清理。
- 图片最多 10 张、文档最多 5 个和总量 50 MB 校验。
- 会话附件工作集 100 MB 校验与历史附件去重。
- 停止、失败、重试、编辑后截断消息。
- 图片模式与文档附件互斥。
- 图片参数摘要、恢复和 Codex 2k/4k 限制。
- 聊天图片持久化会丢弃 `b64_json`，只保存 URL 和元数据。

### 14.3 构建与浏览器验证

- `bun run build`。
- 普通用户登录后默认进入 `/chat`。
- 桌面固定侧栏和移动端历史抽屉。
- 模型选择、Markdown、代码复制、停止、重试和编辑消息。
- 刷新后恢复会话、附件、滚动位置和图片任务。
- 切换普通用户后不可看到其他用户历史。
- 上传图片、PDF、DOCX、XLSX、PPTX、TXT、MD、CSV 的界面状态。
- 生图参数与 `/image` 保持一致，生成和参考图编辑结果进入聊天时间线。

### 14.4 真实上游验证

增加显式 opt-in 的真实附件测试，至少验证：

- 一份 PDF 的连续追问。
- 一份 DOCX 的内容理解。
- 一份 XLSX 或 CSV 的表格理解。
- 图片和文档混合输入。
- 10 张图片输入边界。

文档附件协议探测和至少一份 PDF 的真实请求是该能力的完成门槛。当前环境没有可用真实上游账号时，相关任务应标记为阻塞，最终交付不得声称通用文档附件已经完成；其他模块仍可继续实现和验证。

## 15. 多 Agent 实施与提交策略

当前并发上限为主 agent 加 3 个子 agent。实施时尽可能保持 3 个子 agent 同时工作；主 agent 负责接口契约、worktree 管理、按依赖集成和最终验证。所有子 agent 使用独立 git worktree 和独立分支，避免共享工作区提交相互污染。

每项任务必须先写失败测试、完成实现、运行聚焦验证，然后分别提交。任务提交不 squash。一个任务不得顺手修改另一个任务的文件边界；确有跨模块需要时，由主 agent 在后续集成任务中处理并单独提交。

### 15.1 第一批：后端协议与前端基础并行

1. **上游通用附件适配器**：先完成真实协议探测和脱敏 fixture，再实现通用文件上传、附件元数据、账号内上传缓存及单元测试。没有真实 PDF 验证不得完成任务。
2. **Web 聊天流式接口**：multipart manifest 校验、同账号编排、SSE、断连取消与账号重试规则及 API 测试。该任务只依赖已约定的 `UploadedChatAttachment` 内部接口，可使用测试替身并与任务 1 并行；最终集成必须等待任务 1 的 fixture 与真实验证通过。
3. **前端聊天协议层**：聊天类型、SSE 解析、模型分类、附件校验和对应 Vitest 测试。

### 15.2 第二批：存储、图片参数与表现组件并行

4. **本地会话与附件存储**：`subjectId` 隔离、Blob 去重、引用清理、配额错误及测试。
5. **共享图片设置组件**：从现有 `ImageComposer` 提取参数定义和设置 UI，完成 `/image` 回归测试。
6. **聊天表现组件**：侧栏、消息、Markdown、模型选择、输入框、附件卡片、桌面与移动端组件测试；通过 props 和测试替身对接协议层。

### 15.3 第三批：页面编排、图片任务与应用入口并行

7. **聊天页面编排**：连接 Store、SSE、编辑/重试/停止和消息状态机。
8. **聊天内图片任务**：连接现有生成/编辑任务、完整参数快照、轮询恢复、结果操作及测试。
9. **路由感知应用壳层**：`/chat` 沉浸式布局、普通用户默认路由、导航、权限和 basePath 回归。

### 15.4 第四批：独立验证并行

10. **后端回归与真实附件验证**：运行聚焦 pytest、完整后端测试和 opt-in 上游 PDF/DOCX/XLSX/图片验证。
11. **前端构建与浏览器 QA**：运行前端测试和生产构建，验证桌面、移动端、刷新恢复及多用户隔离。
12. **图片工作台回归与视觉一致性**：验证 `/image` 原有参数、任务恢复、参考图和聊天页共享组件没有视觉或行为回归。

验证任务发现的问题按所有权返回对应 agent；如果原任务已经提交，修复使用独立中文 Conventional Commit。主 agent 最终按依赖顺序汇入所有任务提交，解决必要冲突，运行综合测试，并保留每一个任务和修复提交。

## 16. 验收标准

1. 普通用户登录后进入可用的 `/chat`，管理员功能不回归。
2. 用户可以选择聊天模型并获得可停止的流式 Markdown 回答。
3. 会话支持新建、重命名、删除、编辑消息、复制和重新生成。
4. 图片与指定文档格式可上传，限制和错误明确。
5. 文档由同一个上游账号原生上传并参与连续对话。
6. 生图显式开启并完整复用现有图片模型、质量、尺寸、比例、档位和数量参数。
7. 最多 10 张输入/参考图片可通过前后端验证；现有生图数量能力保持不变。
8. 会话、附件和参数按 `subjectId` 本地持久化且互相隔离。
9. `/image` 的现有生成、编辑、任务恢复和参数行为不回归。
10. 后端测试、前端测试、生产构建和桌面/移动浏览器验证通过。
11. 各实现任务保留独立提交，并能从提交历史识别任务边界。
12. 流内错误和用户停止都是不可被 `[DONE]` 覆盖的终态，用户停止会关闭后端上游资源。
13. 聊天图片历史不持久化 Base64，只保存服务端 URL、task ID 和必要元数据。
